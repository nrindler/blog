Abstract¶

We simulate a two-player dice game in Python to obtain the expected number and distribution of cycles the game will last for (Project prompt #14). The game is a simple two player dice game with the goal being to outlast your opponent and not lose all of your coins. The rules of game as:

  • Two players start with 4 coins each, with two coins in the pot.
  • The players take turns tossing a six-sided die, and depending on the value of the die, each player either adds or subtracts from the coins in the pot, or else takes no action.
  • The game ends when one player is unable to perform the task, i.e., runs out of coins and can no longer add coins to the pot.

We use simulation and first step analysis to describe key characteristics of the game, yielding an expected value and distribution of the number of cycles in each game.

  • We observe a mean of 14.8 and a median of 13 cycles across 1 million simulated replications of the game.
  • We demonstrate that as few as 10,000 replications yields an expected value within 0.5% of the expected value obtained from the full 1 million replications.
  • We use batch means to estimate a 95% confidence interval of 14.8 $\pm$ 0.016 cycles.
  • We use Python to test and select a line from a gamma distribution that best fits the observed distribution from the simulation.
  • We validate the results of our simulation using a Markov Chain first-step analysis, which yields an expected value of 14.6 cycles.
  • Our simulation and first step analysis can be extended for this game given any possible allocation of the starting 10 coins.

Background¶

Simulation is the imitation of the operation of a real-world process or system over time. Simulation helps us describe and analyze system behavior, and to answer 'what if' questions. In this study, we use a simple two-player dice game to demonstrate powerful techniques that can be applied to model much more complicated systems.

Simulation sheds light on how soon the game will typically take to terminate, outputting the expected value (mean) number of cycles in the game as well as a distribution of values that we can later fit a model to. In parallel to simulation, we obtained the expected value for the number of cycles from the game using first-step analysis.

First step analysis is a method for evaluating a Markov Chain, which requires definition:

A Markov chain is a mathematical system that experiences transitions from one state to another according to certain probabilistic rules. The defining characteristic of a Markov chain is that no matter how the process arrived at its present state, the possible future states are fixed. In other words, the probability of transitioning to any particular state is dependent solely on the current state and time elapsed.$^1$

Our dice game is a Markov Chain because each turn (each die roll) is independent of one another. The number of coins that Player A/B/pot have at the end of a turn depends solely on the number of coins they have at the start of the turn. The past history of turns is irrelevant for determining the coint count after the turn. Instead, the value of the random die toss determines the coin counts after the turn, and the probability of each die value is known. Thus we can assign probability $P_{ij}$ that any starting state $i$ leads to any potential next state $j$, and these probabilities can be stored in a transition probability matrix $M$.

What happens when one player loses? The game ends. This is called an absorbing state, and is encoded as the transition probability $P_{ij} = 1$. There are many absorption states and many transition states, depending on the parameters of the Markov Chain. As we show below, this two-player die game contains 91 potential states of which 25 are absorbing states.

The first step analysis helps to answer the question of how long it takes on average to reach one of the absorbing states.$^2$ This is analogous to obtaining the mean number of cycles from a simulation. We can compare the result of the first step analysis to the simulation to see how close the two methods are.

def randgen():
    #return number from 1-6

Next we define an $action()$ function that reflects the potential outcome of each turn, i.e., one die toss. The function takes as inputs the starting coins in the player's cup (turn_count), the number of coins in the pot (coincup_count), and the value of the die (output from the PRN generator above).

Per the rules of the game (see Appendix), If the player rolls a:

  • 1, then the player does nothing.
  • 2: then the player takes all coins in the pot.
  • 3: then the player takes half of the coins in the pot (rounded down).
  • 4,5,6: then the player puts a coin in the pot.

After each turn, the $action()$ function outputs the updated number of coins in the player's cup (turn_count) and the number of coins in the pot (coincup_count).

In [104]:
#define the scoring action, returns the coin count and cup count regardless if it is negative
def action(turn_count,coincup_count,dice):
        if dice==1:
            return(turn_count,coincup_count)
        elif dice==2:
            turn_count=turn_count+coincup_count
            coincup_count=0
            return(turn_count,coincup_count)
        elif dice==3:
            turn_count=turn_count+math.floor(coincup_count/2)
            coincup_count=math.floor(coincup_count/2)+1
            return(turn_count,coincup_count)
        else:
            turn_count=turn_count-1
            coincup_count=coincup_count+1
            return(turn_count,coincup_count)

Finally, we create our main $simulation()$ program. This function takes as inputs the number of coins allocated to Player A (A), Player B (B), and the pot (C) at the start of the game. At each turn, the dice roll is a randomly generated integer from the $randgen()$ function above. Each player's turn occurs by calling the $action()$ function previously defined. After each turn, the number of coins in the player's hand and the pot are updated. If after one full cycle (i.e., both Player A and Player B take their turn) one player has no coins and is required to pay coins into the pot because they rolled a 4/5/6, then the game ends. The total number of cycles (including the final cycle) is appended to the cycle_counter list. The cycle_counter list will have one item for each replication of the simulation.

In [105]:
# Set up counter
cycle_counter = []

def simulation(A,B,C):
    
    #A is the number of coins that A starts with. B is the number of coins that B starts with
    #coincup= number of coins in pot  
    A0=A
    B0=B
    coincup=C
    
    #cycle count start
    cycle=1
    #initialize the floor loop
    test_score=0
    while test_score==0:
        #Roll dice
        dice=randgen()        
        #A turn
        A0,coincup=action(A0,coincup,dice)
        #B turn
        B0,coincup=action(B0,coincup,dice)
        #Check if A or B runs out of coins and loses the tame
        if ((A0< 0) or (B0<0)):
            cycle_counter.append(cycle)
            test_score=1
        cycle=cycle+1

We execute our $simulation()$ function inputting the number of starting coins for Player A, Player B, and the pot (4, 4, and 2, respectively). This function is extendable to allow the analyst to input any (positive integer) starting values for the game. Future analysis could evaluate how sensitive the expected value and distribution of cycles is to the starting coin counts.

Confidence Interval¶

Now that we have the expected value for the number of cycles in a game, we can produce a 95% confidence interval using the batch means method. Our analysis above suggests that a batch size (m) of 10,000 should get us pretty close to the expected value we obtained from the full simulation with 1,000,000 replications. That yields 100 batches (b=100).

The confidence interval is given by $\mu \pm t_{\alpha /2, b-1}\sqrt{\hat{V}_B/n}$

In [111]:
# Get batch means
def batchmeans(l, n):
    n = max(1, n)
    a = (l[i:i+n] for i in range(0, len(l), n))
    b = list(a)
    c = list(map(np.mean,b))
    return (c)

# Set m=10,000 and b=100

How quickly can you lose the game?¶

As an interesting aside, given the rules of the game, we know that the minimum possible number of cycles is 5: each player starts with 4 coins at at worst loses 1 coin per cycle (with a roll of 4, 5, or 6). Thus it takes a minimum of 5 turns for the player to lose by having no coins in their hand and owing one to the pot. This requires 5 consecutive die rolls of 4/5/6. The probability of losing in 5 turns is $0.5^5 = 0.03125$, since a roll of 4/5/6 occurs with a probability of 50% each time. How does this compare with the simulation output?

In this game, absorption states reflect when one player loses, i.e., they end the turn with -1 coins. The following script cycles through each of the 91 potential states. It instantiates a 91x91 matrix of zero's called "Probstates." We loop through each of the 91 states:

  • First, absorption states are identified and written into the Probstates matrix along the diagonal axis to create an identity matrix (row index = column index)
  • Next, for transient states, each starting state has 16 potential next states based on the outcome of one cycle of the game.
    • As we saw in the previous code cell, one turn can have four different actions depending on the value of the die. 4x4 = 16 potential outcomes from one cycle (i.e., two turns).
    • Probabilities are assigned and recorded in the Probstates matrix representing the percentage likelihood that each of the 16 potential next states occur. Recall that since this game is a Markov process, the probability of the next state depends only on the current state and not on any prior states.

Finally, we can output the expected value of the first step analysis. We select the starting point of (4,4,2) representing the number of starting coins for Player A, Player B, and the pot, respectively. We remove the 25 absorption states, leaving us with a 66x66 matrix (Q). Using linear algebra we take the inverse of the result of the identity matrix I minus Q, and then sum the row of matrix Q that corresponds with our starting point (4,4,2). The sum of the probabilities of each combination of states $\Sigma\ P_{ij}$ yields the expected value of the number of cycles in the game.

Q=Probstates[i:,i:]
I=np.identity(Q.shape[0])
W=np.linalg.inv(I-Q) # inverse of identity matrix I minus matrix Q
ev_fsa = W.sum(axis=1)[startQ] # sum up rows of W = total number of expected cycles to reach the absorption state

Conclusion¶

Simulation and first step analysis are powerful tools that help us describe key characteristics of complex systems. This paper uses a simple two-player dice game, but the same methods can be applied to model real-world systems in diverse industries such as manufacturing, supply chain, reliability, and finance.

In summary:

  • We observe a mean of 14.8 and a median of 13 cycles across 1 million simulated replications of the game.
  • We demonstrate that as few as 10,000 replications yields an expected value within 0.5% of the expected value obtained from the full 1 million replications.
  • We use batch means to estimate a 95% confidence interval of 14.8 $\pm$ 0.016 cycles.
  • We use Python to demonstrate that the observed distribution from the simulation is best fit by a gamma distribution.
  • We validate the results of our simulation using a Markov Chain first-step analysis, which yields an expected value of 14.6 cycles.
  • Our simulation and first step analysis can be extended for this game given any possible allocation of the starting 10 coins. -We see that our simulation and first step analysis produce similar results for the smooth and simple cases such as (4,4,2) but create differences when bi-modal or similar distributions result from the simulation. -The more coins in the players hands at the start of the game seems to give better agreement between the first step analysis and the simulation results.

Rules of the Game¶

There are two players, A and B. At the beginning of the game, each starts with 4 coins, and there are 2 coins in the pot. A goes first, then B, then A,. . . . During a particular players turn, the player tosses a 6-sided die. If the player rolls a:</br>

  • 1, then the player does nothing.
  • 2: then the player takes all coins in the pot.
  • 3: then the player takes half of the coins in the pot (rounded down).
  • 4,5,6: then the player puts a coin in the pot.</br>

A player loses (and the game is over) if they are unable to perform the task (i.e., if they have 0 coins and need to place one in the pot). We define a cycle as A and then B completing their turns. The exception is if a player goes out; that is the final cycle (but it still counts as the last cycle).
